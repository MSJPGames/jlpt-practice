# 第1部：サイト保守の指針

## 基本
- 返答は**日本語**で。ユーザーは Henrique。
- サイトは GitHub Pages（https://msjpgames.github.io/jlpt-practice/）で公開。作業ファイルは **OneDrive の `JLPT PRACTICE` フォルダ**にあり、デバイスブリッジ（デバイス名 `enrique-p`）経由で読み書きする。
- 各ページは**自己完結HTML**。共通デザイントークン（`--bg:#F7F5F0; --surface:#FFFFFF; --border:#E2DDD6; --text:#1C1C1C; --text-muted:#6B6560; --text-hint:#9E9890; --radius:14px`、`--accent` は文脈ごと）を守る。音声は端末TTS（`SpeechSynthesis`）＋`[data-say]`＋IIFE方式。
- 各ページの詳細なアーキテクチャ・ファイル構成・データ仕様は、必要なとき `handover.md` を参照（辞書として）。

## 真実の元（重要）
- **本体は常に OneDrive の実ファイル**。プロジェクトの資料はスナップショットなので、**実ファイルの中身は毎回 OneDrive から最新をステージングしてから**編集する。記憶や資料の断片で上書きしない。
- 既存ファイルに追記する作業（例：BJTの増量）は、**上書き前に必ず現在の実データを確認**する。

## ステージングのキャッシュずれ対策
- 取り込んだファイルの**実バイト数**が `device_list_dir` の **size と食い違う**ときは、キャッシュが古い。その版で上書きしない。
- 対処：ユーザーに対象ファイルを**別名コピー**（エクスプローラーで Ctrl+C→Ctrl+V＝「〜 - コピー」）してもらい、新しいパスをステージングして最新を読む。編集後は元の名前で書き戻す。

## 納品ルール
- **GitHubにアップロードすべきファイルが2つ以上なら、必ず1つのZIPにまとめて渡す**（1ファイルだけならそのまま）。ZIP名は内容が分かる名前、中身はGitHubのフォルダ構成どおりの相対パス。
- OneDrive への反映は `device_commit_files`（`force:true`）で**各ファイル個別**に行う。ユーザーに渡すのは基本ZIP。
- 完成物・ドラフトは `SendUserFile` で渡す。

## 新しい言語を追加するとき
- `langdata.js`（一覧・個別ページ）＋`chardata.js`（なぞり書き）＋必要なら`grammardata.js`（文法）に追加。
- **文法データが無いうちは langdata に `grammar` フィールドを付けない**（リンク切れ防止）。moji リンクは常に出るので `chardata.js` の CHARS は必須。
- 解説ページは `extras:[{href,title,desc}]` で各言語ページに接続する。

## 検証
- 非自明な変更は**渡す前に Playwright（Chromium）で検証**する。`file://` でページを開き `pageerror` のみ確認（単体表示の `ERR_FILE_NOT_FOUND` は無害なので無視）。データファイルがパースできること、リンク・項目数・表示を確認する。

## 進め方
- 作業はテーマごとにチャットを分けると context が保ちやすい（例：「聴解の追加」「新言語ページ」「文法データ」）。
- 旧世代の重複フォルダ（`jlpt-site/`・古い `games/` 内の重複）は指定がない限り触らない。現行版は原則 root 直下の新しいファイル。
- スコープが曖昧なときは確認する。ただし返答が難しい状況では、妥当な解釈を明示して進める。

# 第2部：問題作成の規約（クイズ問題を作る・増やすとき）

> クイズ問題（JLPT等）を作成・増量するときは、必ずこの規約に従う。ルールを決め直したら、ここを更新すれば以後すべての問題に反映される。

## 制作方針（前提）
このサイトはJLPTを中心に、**N5〜N1**の学習者（独学者・日本語学校の自習用）に向けた問題を提供する。指示文は「**やさしい日本語＋英語**」を基本とする。問題を作るときは次の4原則を守る。

1. **一貫性を最優先**：同じ種類の教材は必ず同じ形式で作る（＝この規約に従う）。
2. **JLPT公式形式に準拠**：実際の試験の問題形式・出題順に合わせる。
3. **必ず解答＋簡単な解説をつける**：独学者が自分で復習できるように。
4. **段階的に拡充**：完成したレベルを基準に、他レベルへ横展開する。

## 使用方針（著作権）
- 公式の問題例・過去問・解説資料は、**形式・難易度・出題の型を学ぶための「お手本（参考）」としてのみ**使う。
- 公開サイトに載せるのは、**そのお手本に沿って新しく作ったオリジナル問題のみ**。公式問題文の転載はしない。

## 全体の共通ルール
- 選択肢は**4つ（1・2・3・4）**。
- 正解は**必ず1つ**。誤答（ディストラクター）は「もっともらしいが明確に間違い」であること。
- 各大問の冒頭に**指示文**を置く（日本語＋やさしい英語）。
- 問題本体と**解答・解説は必ず分ける**（解答は末尾にまとめる）。
- 解説は1問1〜2行。「なぜ正解か／他がなぜ違うか」を簡潔に。
- ルビ（ふりがな）は原則つけない。ただしN1でも難読の固有名詞などは例外可。
- 1ファイル＝1模試の1セクション。ファイル名は `レベル_セクション名` で統一。

## 聴解問題の共通ルール（音声）
聴解問題を作るときは、テストの種類（**JLPT・J.TEST・BJT など**）を問わず、次を守る。

- **指示文（ナレーション）の声は、会話（登場人物）の声と必ず別にする。** 状況説明や質問文を読む「ナレーター」の声を、会話する男性・女性の声と同じにしない。聞いたときに紛らわしくならないようにするため。
- 声の割り当ての目安（edge-tts の例）：ナレーター＝`ja-JP-DaichiNeural`（落ち着いた男性）／会話の女＝`ja-JP-NanamiNeural`／会話の男＝`ja-JP-KeitaNeural`。使える声が2種類しかない環境では、ナレーターを低め・ゆっくりに調整するなどして、**必ず会話と区別**する。
- 音声は1問につき1つのmp3にまとめる（状況→会話→質問 の順など）。`audio/` に置き、データの `audio` フィールドにファイル名をひもづける（例：`audio/n3_q1_01.mp3`）。話者つきの**スクリプト**も持たせ、画面で開閉表示できるようにする。
- 音声生成は各レベルの `make_*_audio.py`＋`.bat`（edge-tts、必要なら pydub＋imageio-ffmpeg で連結）で、PC 側で行う。

### 出題形式メモ（公式問題例より・N1文法）
- **問題5（10問）**：単なる文法穴埋めではなく、「お知らせ」「求人サイトのQ&A」「インタビュー」「会話」など**現実的な場面設定**が多い。**敬語（尊敬・謙譲）の使い分け**や、微妙な**文末表現・ムード**も問う。選択肢は短い語のことも、長めの文末表現のこともある。
- **問題6（5問）**：4つの空所のうち **★の位置に入る語**を選ぶ並べ替え。設問群の冒頭に「問題例＋解答のしかた」を置く。
- **問題7（5問）**：500〜1000字程度の**文章（エッセイ等）に5つの空所**。接続表現・指示・文末など、**文章の流れ**に合う選択肢を選ぶ。

> 📝 各レベル（N5〜N1）で大問構成は少しずつ異なる。新レベルに着手するときは、その級の公式「大問のねらい」をこの規約に追記してから作ること。

## 記法ルール（Markdown）
- 下線で問う語は **太字** で示す（例：事態を**把握**する）。
- 空所は `（　　）` （全角スペース）で表す。
- 大問見出しは `## 問題N ○○`。
- 各設問は番号付きリスト、選択肢は `1. 2. 3. 4.`。
- 解答欄の見出しは `# 解答・解説` とし、**必ず改ページ相当の区切り**（`---`）を入れる。
